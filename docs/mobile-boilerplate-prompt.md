# 모바일 앱 풀스택 보일러플레이트 제작 프롬프트

> 이 문서는 웹 보일러플레이트(이 레포지토리)와 같은 철학으로 **모바일 앱 + 서버
> 모노레포 보일러플레이트**를 Claude Code에게 요청하기 위한 프롬프트입니다.
> 새 레포지토리에서 아래 내용을 그대로 명령으로 사용하세요.

---

모바일 앱 서비스 개발을 위한 보일러플레이트를 만들어 주세요. 서버와 모바일 클라이언트가
같은 레포지토리 안에 있는 모노레포 구조를 원합니다.

사용해야 하는 기술: React Native(Expo 기반), TypeScript, Bun(패키지 매니저·스크립트
러너·서버 런타임), Bun server

미리 알아 둘 전제: 앱 런타임은 Metro/Hermes이므로 Bun은 앱의 런타임이 아니라 패키지
매니저·서버 런타임으로 사용합니다. HMR은 앱은 Metro Fast Refresh, 서버는 Bun watch로
각각 적용해 주세요.

## 공통 요청 사항

- 워크스페이스 구조는 `apps/server`, `apps/mobile`, `packages/shared`를 기본으로 해
  주세요. shared는 순수 TypeScript로만 작성해서 Metro와 Bun 양쪽 번들러가 모두 소비할
  수 있어야 하고, 플랫폼(Node/브라우저/네이티브) 의존 코드가 들어가면 안 됩니다.
- 서버와 클라이언트 공통으로 사용하는 것(검증 스키마, 도메인 타입, API 규약, i18n
  카탈로그, 시간 유틸 등)은 shared에 작성해 주세요.
- 유틸리티는 구현이 바뀔 수도 있습니다. 견고하고 유연한 인터페이스를 중시하고, 구현
  세부 사항은 파사드 형태로 하여 외부에서 신경쓰지 않아도 되게 해 주세요.
- 의존 방향: 서버는 클라이언트의 모듈을 사용해도 괜찮지만, 클라이언트는 서버의 런타임
  모듈을 참조할 수 없게 해 주세요. 타입 전용 import는 허용합니다. 이 규칙은 ESLint로
  강제해 주세요.
- schema validation을 적용해 주세요. 우선 zod/mini를 기본으로 하되, 나중에 yup 같은
  것으로 바꿔도 동작하도록 라이브러리 중립적인 `Validator<T>` 파사드 뒤에 숨겨 주세요.
- i18n에 대응해 주세요. 기기 로케일 자동 감지 + 앱 내 언어 변경을 모두 지원하고,
  서버 에러 메시지도 로케일 협상(`Accept-Language`)으로 로컬라이즈해 주세요.
- 시간은 서버 내부에서 항상 UTC로 저장/처리하고(DB는 `timestamptz`), 타임존 변환은
  경계(클라이언트 표시 시점, 기기 타임존 사용)에서만 수행해 주세요.
- 목록 API는 반드시 페이지네이션을 지원해야 합니다. 모바일은 무한 스크롤이 기본이므로
  **커서 기반 페이지네이션을 기본**으로 하고, 정렬·필터는 일관된 파라미터 규약을
  따라야 합니다. TanStack Query의 `useInfiniteQuery`와 자연스럽게 맞물려야 합니다.
- 환경별 설정(로컬/개발/운영)은 코드 수정 없이 전환 한 번으로 바뀌어야 합니다.
  - 서버: 환경 변수(`.env`) 교체 한 번. 부팅 시 스키마로 검증.
  - 앱: 빌드 프로필(dev/staging/prod)별로 API base URL·번들 ID·앱 이름이 달라져서
    한 기기에 환경별 앱을 나란히 설치할 수 있어야 합니다(Expo app config의 동적 설정 사용).
  - 비밀값은 절대 저장소에 커밋되면 안 됩니다. 특히 **앱 번들은 공개물**이라는 전제로,
    앱에 포함되는 값에는 어떤 비밀도 넣지 마세요(비밀이 필요한 일은 전부 서버가 대행).
- 테스트를 작성해 주세요. 비즈니스 로직은 단위 테스트, API는 통합 테스트, 앱의 핵심
  로직(버전 비교, 부트 시퀀스 상태 머신, 원격 설정 파싱 등)도 단위 테스트로 검증하고,
  주요 성공/실패 케이스를 모두 포함해 주세요. 테스트는 외부 환경 없이 명령 한 번으로
  실행 가능해야 합니다(인메모리 DB 사용).

## 앱 버전·업데이트 정책 (가장 중요한 요구사항)

- **서버가 버전 정책의 단일 진실 공급원**입니다. 플랫폼(ios/android)별로
  `minSupportedVersion`, `latestVersion`, 업데이트 방식, 안내 메시지를 담는 DB 테이블과
  이를 반환하는 API를 만들어 주세요. 앱은 시작 시 + 포그라운드 복귀 시 이 정책을
  확인합니다.
- 업데이트는 세 가지 경로가 있고, 정책 응답에 따라 앱이 알맞은 UI를 보여줘야 합니다:
  1. **OTA 업데이트(JS 번들만 변경)**: 버튼 한 번으로 다운로드 → 재시작까지 완료.
     expo-updates를 사용하되, 업데이트 배포 채널은 파사드로 추상화해서 EAS Update와
     셀프호스팅 업데이트 서버 어느 쪽으로도 갈 수 있게 해 주세요.
  2. **스토어 업데이트(네이티브 변경)**: 버튼을 누르면 App Store / Google Play로 이동.
     Android는 in-app update API를 쓸 수 있도록 자리를 마련해 주세요.
  3. **강제 업데이트**: 앱 버전이 `minSupportedVersion` 미만이면 전체 화면을 차단하고
     업데이트 버튼 외에는 아무것도 못 하게 해 주세요.
- 강제가 아닌 업데이트는 "나중에" 스킵이 가능하고, 스킵한 버전은 일정 기간 다시
  묻지 않는 정책도 구현해 주세요.
- **runtimeVersion 개념을 명확히** 해 주세요: OTA는 같은 네이티브 runtimeVersion 안에서만
  안전합니다. 어떤 변경이 OTA로 나갈 수 있고 어떤 변경이 스토어 배포를 요구하는지
  판단 기준을 문서로 남겨 주세요.
- **버전 스큐 대응**: 웹과 달리 모바일 클라이언트는 강제 새로고침이 불가능합니다.
  따라서 서버 API는 구버전 앱과의 하위호환을 유지해야 합니다(응답 필드 제거·의미 변경
  금지 규약을 문서화). 앱은 모든 요청에 `X-App-Version`/`X-Platform` 헤더를 보내고,
  서버는 이를 근거로 지원 종료된 버전에 426(Upgrade Required) 류의 명확한 응답을 주며,
  breaking change는 오직 `minSupportedVersion`을 올려 강제 업데이트로만 해소합니다.

## 지원 기기·플랫폼 분기 정책

- **삼성(Android)과 애플(iOS) 기기만 지원**합니다. 그 외 기기·플랫폼(웹, 태블릿 최적화,
  폴더블 특수 대응 등)을 위한 코드와 설정은 과감히 배제해 주세요. 지원 매트릭스(최소
  OS 버전, 대상 기기군)를 문서에 명시해 주세요.
- 플랫폼별 분기는 다음 원칙을 따라 주세요:
  - 통합할 수 있는 부분은 과감히 통합(공통 컴포넌트 + `Platform.select` 최소화).
  - 통합이 애매한 부분은 **코드 재사용성을 희생하더라도 과감히 분리**
    (`Component.ios.tsx` / `Component.android.tsx` 파일 분리 컨벤션).
  - 분기가 생길 때마다 왜 분리했는지 반드시 기록하는 규약을 만들어 주세요:
    코드 주석 + `docs/platform-decisions.md` 한 문서에 결정 이력 축적.
  - 이 컨벤션을 보여주는 실제 분리 예시를 하나 이상 포함해 주세요.

## 앱 시동 화면 (스플래시 / 부트 시퀀스)

- 네이티브 스플래시 → JS 부트 화면 → 메인 화면으로 이어지는 부트 시퀀스를 **명시적인
  상태 머신**으로 구현해 주세요. 각 단계가 코드에서 명확히 드러나야 합니다.
- 부트 중에 수행할 일: 원격 설정 로드, 버전 정책 확인(강제 업데이트/점검 모드면 여기서
  차단), 세션 복원 자리.
- 시동 화면에 **광고가 들어갈 수 있습니다**. 실제 광고 SDK는 붙이지 말고, 광고 슬롯
  인터페이스(로드/노출/스킵/실패 콜백)와 더미 구현을 만들어 주세요. 최소 노출 시간과
  스킵 허용 여부는 원격 설정으로 제어할 수 있어야 합니다. 광고 로드가 실패하거나
  느려도 앱 진입이 막히면 안 됩니다(타임아웃 필수).
- 시동 화면의 디자인 교체가 쉬워야 합니다 — 컴포넌트 하나를 갈아끼우면 되는 구조로.

## 원격 설정 (Remote Config)

- 앱 설정을 담는 `app_config` DB 테이블을 만들어 주세요. key-value + revision(또는
  updated_at) 구조로 하되, 나중에 플랫폼별·앱 버전별 오버라이드를 넣을 수 있게 스키마를
  확장 가능하게 설계해 주세요.
- 설정 변경이 앱에 빠르게 반영되어야 합니다. 두 경로를 모두 구현해 주세요:
  1. **폴링**: 간격은 설정 가능(기본 1분), revision/ETag 비교로 변경 없으면 본문 없이
     응답(304)해 트래픽을 아껴 주세요.
  2. **서버 push**: WebSocket으로 설정 변경을 브로드캐스트. 서버가 다중 인스턴스일 수
     있으므로 pub/sub(기본 memory, redis 전환 가능)을 경유해 팬아웃해 주세요. 앱은
     연결 중엔 push로 받고, 백그라운드 복귀 시 폴링으로 보완합니다.
- 앱에서의 소비는 `useConfig()` 훅 하나로 통일해 주세요.
- 이 테이블로 제어되는 데모를 포함해 주세요: 기능 플래그, 공지 배너, 그리고
  **점검 모드(kill switch)** — 서버에서 점검 플래그를 켜면 앱 전체가 점검 화면으로
  전환되어야 합니다.

## 클라이언트(앱) 요청 사항

- 모든 API는 한 곳에서만 관리되고, 처음 보는 개발자도 어떤 API인지 잘 알 수 있도록
  주석을 잘 달아 주세요. 컴포넌트는 fetch를 직접 부르지 않습니다.
- tanstack query로 서버 데이터를 관리하고 로딩/에러(재시도)/빈 상태/정상 데이터 처리를
  모두 구현해 주세요. 목록은 `useInfiniteQuery` 무한 스크롤로.
- tanstack mutation으로 변경을 요청하고 필요한 캐시를 invalidate해 주세요. 적절한
  시점에 optimistic update(스냅샷 → 즉시 반영 → 실패 롤백 → 재동기화)를 사용해 주세요.
- 최소한의 상태만 사용해 주세요. 어떤 상태를 만들면 그것이 derive될 수 있는 게 아닌지
  항상 다시 생각해 주세요. 화면 파라미터는 내비게이션 params에서 파생시키세요.
- 내비게이션은 당신이 원하는 라이브러리(React Navigation 또는 Expo Router)를 쓰되,
  **딥링크(custom scheme + iOS Universal Links / Android App Links)** 대응 구조를
  갖춰 주세요. 알림 탭 → 특정 화면 이동이 가능해야 합니다.
- 디자인 시스템: 원하는 색상으로 디자인 토큰과 컴포넌트를 만들어 주세요. 다크/라이트
  테마(기기 설정 따르기 + 수동 토글)와, 서로 모습이 다른 A 디자인(심미성)과 B
  디자인(시인성, 큰 글씨·고대비)을 설정 한 번으로 스위칭할 수 있어야 합니다.
  디자인 시스템 전체를 한눈에 보는 **앱 내 화면**이 필요합니다(개발 빌드에서 접근).
  아이콘 라이브러리는 당신이 원하는 것으로.
- 접근성: 웹의 WAI-ARIA에 해당하는 React Native accessibility props
  (`accessibilityRole`/`Label`/`State`/`Hint`)를 전 컴포넌트에 준수하고, VoiceOver와
  TalkBack을 염두에 두세요.
- UI 자동화: 모든 인터랙티브 요소는 `testID`가 필수 prop이고 값은 중앙 레지스트리에서만
  나오게 해 주세요. E2E는 Maestro(권장) 또는 Detox로 핵심 플로우 시나리오를 몇 개
  작성하고, 규약 전체를 `docs/ui-automation.md` 한 문서에서 확인할 수 있게 해 주세요.
- 오프라인/네트워크: 네트워크 상태를 감지해 오프라인 배너를 띄우고, TanStack Query
  캐시를 디스크에 persist해서 재실행 시 마지막 데이터를 즉시 보여주게 해 주세요.
  실패한 요청의 재시도 정책을 명시적으로 정의해 주세요.
- 보안 저장소: 토큰 등 민감한 값은 Keychain/Keystore(expo-secure-store) 파사드 뒤에
  저장하세요. 본격적인 인증 구현은 스코프에 넣지 말되, API 클라이언트에 인증 헤더 주입
  훅과 토큰 저장·갱신 자리를 마련해 두세요.
- 앱 라이프사이클: 백그라운드 → 포그라운드 복귀 시 stale 데이터 refetch, 버전 정책과
  원격 설정 재확인이 일어나야 합니다.
- 크래시 리포팅·애널리틱스는 파사드(인터페이스 + 콘솔 더미 구현)만 만들어 두세요.
  나중에 Sentry 등을 어댑터 하나로 붙일 수 있어야 합니다.
- 랜덤 이미지가 필요하면 picsum.photos/{WIDTH}/{HEIGHT}를 사용하세요.

## 서버 요청 사항

- 서버 수평 확장에 대비해 주세요. 다중 인스턴스 간 통신은 pub/sub 추상화(기본 memory,
  `PUBSUB_DRIVER=redis` 전환)로 해 주세요.
- 서버가 워커 서버 역할을 할 수도 있습니다(`SERVER_ROLE=web|worker|all` 같은 방식).
- 특정 서비스는 싱글톤으로 관리될 수 있으니 컴포지션 루트(컨테이너)에서 프로세스당
  싱글톤을 lazy하게 resolve하는 구조로 대비해 주세요.
- DB는 postgresql을 기본으로 하되, 리포지토리 인터페이스 뒤에 두어 쉽게 교체 가능하게
  해 주세요. 테스트·로컬용 in-memory 구현이 같은 계약을 따라야 합니다.
- **푸시 알림 발송 파사드**를 만들어 주세요(FCM/APNs — Expo push 사용 가능). 실제
  자격증명 없이 dry-run 로깅으로 동작하는 구현을 기본으로 하고, 디바이스 푸시 토큰
  등록/해제 테이블과 API를 포함해 주세요. 원격 설정 변경 알림이나 마케팅 푸시가 여기로
  나갈 수 있습니다.
- CORS는 명시적으로 관리해 주세요. 네이티브 앱은 CORS의 영향을 받지 않지만, 추후
  관리자 웹 콘솔 등을 위해 허용 origin을 환경 설정으로 제어해 주세요.
- graceful shutdown을 지원해 주세요. 종료 신호를 받으면 readiness를 먼저 내리고 처리
  중인 요청·WebSocket을 정리한 뒤 종료해야 합니다.
- N+1 쿼리가 발생하지 않도록 주의하고, 발생 가능한 지점에는 해결 방식을 주석으로
  남겨 주세요.
- 헬스체크 엔드포인트를 제공해 주세요(프로세스 생존 여부와 DB 연결 여부 구분).
- 데이터 일관성이 필요한 작업은 트랜잭션으로 묶고, 트랜잭션 경계가 코드에서 명확히
  드러나야 합니다.
- DB 스키마는 마이그레이션 도구로 버전 관리해 주세요. 새 개발자가 clone 후 명령 한
  번으로 최신 스키마 DB를 만들 수 있어야 하고, 개발용 시드 데이터(버전 정책·원격 설정
  기본값 포함)도 준비해 주세요.
- graphql을 사용할 경우 persisted query 방식으로, 앱에 정의된 쿼리의 핑거프린트만 일반
  사용자에게 허용하고 임의 쿼리는 관리자 토큰에만 허용해 주세요.
- 웹소켓이 사용됩니다(원격 설정 push). 재연결(백오프)과 다중 인스턴스 팬아웃을 고려해
  주세요.

## CI / CD / DX 요청 사항

- prettier, eslint, tsc, knip(안 쓰는 익스포트 체크)를 엄격히 해 주세요. 워크스페이스
  전체에 일관되게 적용되어야 합니다.
- GitHub Actions CI를 만들어 주세요. push마다 lint/typecheck/test/knip이 돌아야 합니다.
  앱의 네이티브 빌드는 CI에서 무거우므로 분리하세요: 매 push에는 JS 레벨 검증만,
  네이티브 빌드(EAS Build 등)는 수동 트리거 또는 태그 기반의 별도 워크플로로 자리만
  마련해 주세요(자격증명이 없어도 워크플로 파일과 문서는 완성되어야 합니다).
- husky 등으로 pre-commit(staged lint/format), pre-push(전체 게이트) 훅을 걸어 주세요.
- 서버는 Bun 번들러로 빌드하고, Docker로 DB 인스턴스 기동과 컨테이너 내 서버 빌드가
  가능해야 합니다.
- 앱 버전 규약: semver + 플랫폼별 빌드 넘버 자동 증가 방식을 정하고 문서화해 주세요.
  코드사이닝 자산(keystore, 인증서, 프로비저닝 프로파일)은 **절대 커밋 금지** —
  gitignore 처리하고 관리 방법을 문서로 남겨 주세요.
- 로컬 개발 연결 문서화: iOS 시뮬레이터(localhost), Android 에뮬레이터(10.0.2.2),
  실기기(LAN IP) 각각에서 로컬 서버에 접속하는 방법을 문서화하고, API base URL은
  환경 설정으로만 제어해 주세요.
- 명령 한 번의 온보딩: `bun install` → `bun run db:setup` → 서버 dev + 앱 dev를 각각
  한 명령으로 띄울 수 있어야 하고, README에 처음부터 끝까지의 흐름을 적어 주세요.
- 릴리즈 플레이북 문서를 작성해 주세요: "이 변경은 OTA로 나가는가, 스토어 배포인가,
  강제 업데이트가 필요한가"를 판단하는 체크리스트와 각 경로의 실행 절차.
